## 摘要
前沿推理调优语言模型在深层演绎任务上仍会失败，通过扩展内部推理来提升性能的成本增长迅速。符号委派提供了一条互补路径：语言模型翻译问题，而求解器执行推理。然而，当前用于逻辑编程的自动形式化流程通常是针对特定任务或智能体定制的集成方案。我们提出了PrologMCP，一个与任务无关的开源服务器，通过模型上下文协议（MCP）将Prolog暴露为有状态工具。其紧凑的工具接口、结构化的错误报告以及每会话隔离机制，使得翻译-运行-检查-修复循环成为支持MCP的智能体的可复用原语。我们在PARARULE-Plus的两个子集上评估了增强PrologMCP的形式化智能体与标准及推理LLM（Claude Sonnet 4.6、GPT-4.1和o4-mini）的性能：一个通用样本，以及一个针对自然语言推理特定失败模式的更具挑战性的样本。在通用样本上，形式化智能体匹配或超越了推理LLM（准确率1.00对比1.00/0.998），相比标准模型（GPT-4.1为0.762）提升最大。在挑战性子集上，形式化智能体保持近乎完美（1.00/0.99），而推理LLM下降至0.95/0.94。这些结果表明，通过MCP将推理委派给Prolog是扩展自然语言推理的一种稳健且可检查的替代方案。

## Abstract
Frontier reasoning-tuned language models still fail on deductive tasks at depth, and the cost of improved performance through extended internal reasoning scales poorly. Symbolic delegation offers a complementary route: a language model translates the problem, while a solver performs the inference. However, current autoformalization pipelines for logic programming are typically bespoke integrations tied to particular tasks or agents. We introduce PrologMCP, a task-agnostic, open-source server that exposes Prolog as a stateful tool through the Model Context Protocol (MCP). Its compact tool interface, structured error reporting, and per-session isolation make the translate-run-inspect-repair loop a reusable primitive for MCP-capable agents. We evaluate a formalizer agent enhanced with PrologMCP against standard and reasoning LLMs (Claude Sonnet 4.6, GPT-4.1, and o4-mini) on two subsets of PARARULE-Plus: a general-purpose sample and a more challenging one targeting a specific failure mode of natural-language reasoning. On the general sample, the formalizer matches or exceeds reasoning LLMs (accuracy 1.00 vs.\ 1.00 / 0.998), with the largest gains over standard models (0.762 for GPT-4.1). On the challenging subset, the formalizer remains near-perfect (1.00 / 0.99) while reasoning LLMs drop to 0.95 / 0.94. These results suggest that delegating inference to Prolog via MCP is a robust and inspectable alternative to extended natural-language reasoning.

## 2. 论文提出的方法论
- **核心思想**：使用MCP协议作为标准化接口，LLM智能体通过MCP工具调用Prolog服务器，将自然语言问题自动形式化为Prolog程序并执行查询，利用结构化反馈迭代修正。
- **关键技术细节**：
  - **PrologMCP架构**：三层设计：
    - **Python MCP服务器**：呈现工具接口，处理发现、参数验证、调度，返回结构化结果。
    - **会话注册表**：管理多个Prolog会话（每会话独立swipl子进程），支持TTL过期、LRU淘汰，线程安全。
    - **Prolog Harness (harness.pl)**：在每个会话子进程中运行，监听stdin上的JSON命令，执行Prolog操作并返回JSON响应。
  - **工具接口**：8个操作工具 + 1个管理工具，包括：
    - `consult_text`：加载Prolog源代码创建会话。
    - `run_goal`：执行查询，支持深度限制、解决方案数限制。
    - `inspect_predicate`：检查谓词定义。
    - `replace_predicate`：替换谓词子句（修复故障）。
    - `trace_goal`：通过元解释器返回证明树（可检查性）。
    - 其他：`get_source`, `list_messages`, `run_tests`, `close_session`。
  - **错误处理**：编译错误非致命，返回结构化diagnostics（severity, kind, message, line），允许智能体修复后重试。
  - **会话隔离**：每个会话使用独立模块前缀，避免命名冲突；子进程资源受限。
- **形式化智能体流程**：
  1. LLM接收自然语言上下文和问题，输出`<prolog>...</prolog>`（程序）和`<goal_true>...</goal_true>`（查询）。
  2. 通过`consult_text`加载程序，若解析/加载失败则返回错误给LLM。
  3. 通过`run_goal`执行查询，获得结果（true/false/超时/深度超限）。
  4. 若出错或结果不确定，LLM可调用`inspect_predicate`、`replace_predicate`等工具修复，最多重试5次。
  5. 最终输出布尔答案。

## 7. 优点
- **标准化接口**：利用MCP协议，与任何MCP兼容的智能体框架集成，无需定制代码。
- **任务无关可复用**：工具设计通用，不仅限于当前数据集，可应用于其他逻辑推理问题。
- **结构化错误报告**：返回类型化的JSON诊断信息，智能体可基于精确位置和类型修复，而非仅原始错误文本。
- **会话隔离与可检查性**：每个会话独立，支持`trace_goal`返回证明树，提供推理透明度和可调试性。
- **实验设计细致**：专门构造SLDNF分歧挑战集，直接针对模型失败模式，强化了结论的可靠性。
- **性能权衡分析**：全面报告token和时间，量化说明符号委派在可靠性与效率之间的优势。
