## 摘要
DeQL（决策查询语言）扩展了SQL以表达决策查询：给定从关系数据中提取的选项、政策约束和可衡量的目标，DeQL查询计算出最佳行动方案。两个构造体承载了这一扩展：CREATE CANDIDATES（创建候选），定义了从关系源中提取的选项空间；以及DECIDE（决策），声明决策变量、命名约束和目标。该设计遵循SQL的原则：用户陈述要优化的内容，而引擎选择解决方法；每个查询消耗并产生关系；问题的结构对引擎保持可见。本文档规定了该语言（设计原则、语法、形式化语法和执行模型），并提供了涵盖子集选择、分配、指派、调度以及多个聚合层级决策的示例，以及用于不确定性下的优化、内联模型评分和时间与质量界限求解的扩展。这是规范的第一版；该语言正在积极开发中，此版本确定了后续修订将基于的核心构造。

## Abstract
DeQL (Decision Query Language) extends SQL to express decision queries: given options drawn from relational data, constraints from policy, and a measurable objective, a DeQL query computes the best course of action. Two constructs carry the extension: CREATE CANDIDATES, which defines the space of options from relational sources, and DECIDE, which declares decision variables, named constraints, and an objective over them. The design follows SQL's principles: the user states what to optimize while the engine chooses how to solve it, every query consumes and produces relations, and the structure of a problem stays visible to the engine. This document specifies the language (its design principles, syntax, formal grammar, and execution model) with examples spanning subset selection, allocation, assignment, scheduling, and decisions at multiple levels of aggregation, and extensions for optimization under uncertainty, inline model scoring, and time- and quality-bounded solving. It is the first version of the specification; the language is under active development, and this version fixes the core constructs on which later revisions will build.

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：传统SQL能够回答“发生了什么”（描述性查询）和“将要发生什么”（预测性查询），但无法直接回答“应该怎么做”（规定性查询）。例如：在预算下选择哪些商品、如何分配GPU小时、如何调度任务等决策问题，通常需要将数据导出到独立的求解器或建模工具中手动编码，过程繁琐、易错且缺乏声明性。
- **研究动机**：希望将决策优化融入数据库系统，遵循SQL的声明式原则——用户只需说明“要优化什么”，引擎自动决定“如何求解”。
- **整体含义**：DeQL是一种扩展SQL的决策查询语言，支持从关系数据中定义候选选项、声明决策变量和约束、优化目标，并返回关系结果。该语言是首个版本规范，为后续实际应用和扩展奠定基础。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：通过两个新的SQL构造体来声明决策问题。
  - `CREATE CANDIDATES`：定义决策的选项空间。它指定一个`DECISION KEY`（类似于主键），声明每个决策实体，数据源自SQL查询。可选的`FOREIGN DECISION KEY`、`DECISION COLUMNS`（默认决策变量）、`CONSTRAINTS`（模式级约束）等。
  - `DECIDE`：执行决策查询。指定候选集（可连接多个候选表）、决策变量（类型包括`BINARY`、`INTEGER`、`CONTINUOUS`，支持`SELECTION`修饰符自动应用变量与数据列的乘法及边界链接）、`WHERE`过滤条件、`SUBJECT TO`约束（包括聚合约束、逐行约束、窗口约束、调度约束`DISJUNCTIVE`和`CUMULATIVE`）、以及目标函数（`MINIMIZE`/`MAXIMIZE`，可选`EXPECTED`、`WITHIN`、`TIMEOUT`）。

- **关键技术细节**：
  - **决策列粒度（Grain）**：默认按`DECISION KEY`每行一个变量，但可通过`BY`子句聚合到更粗的粒度（如按区域），实现多层级决策。
  - **约束中的分组与过滤**：`BY`子句生成每个组一个约束实例；`FILTER`子句限制参与聚合的候选行。
  - **窗口约束**：利用SQL窗口函数（`LAG`、`LEAD`、`FIRST_VALUE`、`LAST_VALUE`）建立决策变量在有序序列中的关系。
  - **多粒度决策**：支持连接多个候选表（如批次调度中，`item_batch_assignments`和`batch_timing`），决策变量可以分属不同候选表，通过`ON`子句指定归属。
  - **扩展**：随机约束（`STOCHASTIC`声明分布，`WITH PROBABILITY`设定机会约束，`EXPECTED`优化期望）、内联模型评分（`PREDICT`调用注册的ML模型）、质量修饰符（`WITHIN`近似保证、`TIMEOUT`时间限制）。
  - **执行模型**：三阶段中间表示（抽象IR、具体IR、物化IR），引擎自动选择求解算法（如线性规划、网络流、混合整数规划等）。

## 7. 优点

- **声明式与SQL兼容**：用户只需描述“做什么”，引擎自动处理“怎么做”；沿用SQL的语法和语义，降低学习成本。
- **关系封闭性**：决策查询输入和输出都是关系，结果可被进一步SQL查询、连接或作为下一个决策的输入，易于集成到现有数据管线中。
- **结构保持**：决策问题的结构（实体、分组、约束范围）对引擎可见，便于识别特殊结构（如总幺模性）并使用专用高效算法（如网络单纯形法）。
- **丰富表达能力**：支持连续/整数/二进制变量、分组/过滤/窗口约束、多粒度联合决策、调度约束（`DISJUNCTIVE`、`CUMULATIVE`）、随机优化、内联ML评分等，覆盖了大部分经典运筹问题。
- **可扩展性考虑**：通过候选抽象（`CREATE ABSTRACTION`）和近似求解修饰符（`WITHIN`、`TIMEOUT`）支持大规模问题的高效近似求解。
