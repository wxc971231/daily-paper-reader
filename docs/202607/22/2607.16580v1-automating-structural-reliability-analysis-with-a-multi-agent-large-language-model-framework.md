## 摘要
结构可靠性分析支持建筑物和土木基础设施的设计与安全评估，但在整个工作流程中需要专业知识。本研究提出了一种多智能体大语言模型框架，可自动化从自然语言问题描述到可靠性指标和失效概率的解释性估计的组件级可靠性分析。专门的智能体处理问题表述、方法规划、代码生成、执行和结果解释，并在关键决策点由人工确认。方法规划器使用QLoRA进行微调，用于先验可靠性方法类别选择。分析结果并非直接由大语言模型生成，而是通过经过验证的确定性求解器计算可靠性估计，从而提高可重复性并降低幻觉风险。该框架使用开放权重模型，支持本地执行，无需封闭API。结果表明，它在保持计算可信度的同时，降低了结构可靠性评估的专业门槛。

## Abstract
Structural reliability analysis supports the design and safety assessment of buildings and civil infrastructure but requires specialized expertise throughout the workflow. This study presents a multi-agent large language model framework that automates component-level reliability analysis from a natural-language problem statement to interpreted estimates of the reliability index and failure probability. Specialized agents handle problem formulation, method planning, code generation, execution, and result interpretation, with human confirmation at key decision points. The Method Planner is fine-tuned using QLoRA for a priori reliability-method category selection. Analysis results are not generated directly by an LLM; instead, validated deterministic solvers compute the reliability estimates, improving reproducibility and reducing hallucination risk. The framework uses open-weight models and supports local execution without closed APIs. Results show that it lowers the expertise barrier to structural reliability assessment while preserving computational trustworthiness.

## 2. 论文提出的方法论

*   **核心思想**：构建一个多智能体大语言模型（Multi-agent LLM）框架，将复杂的可靠性分析工作流分解为多个子任务，由专门的LLM智能体处理，而关键的数值计算（β、Pf）则委托给确定性求解器，避免LLM直接产生数值幻觉。
*   **框架架构（六个组件）**：
    1.  **Orchestrator（编排器）**：用户入口，对用户意图（新问题、修改问题、解释结果等）进行路由。
    2.  **Problem Analyst（问题分析师）**：将用户自然语言输入转换为结构化的 `ProblemSpec` 数据合约（提取随机变量、分布、极限状态函数、相关性）。
    3.  **Method Planner（方法规划器）**：接收 `ProblemSpec`，从 FORM-based、MCS、Advanced 三个类别中先验地选择一个最合适的方法类别。**这是唯一经过微调的智能体**。
    4.  **Code Engineer（代码工程师）**：根据 `ProblemSpec` 和选定的方法，自动生成调用后端求解器的可执行 Python 代码。
    5.  **Runner（运行器）**：非LLM的确定性执行组件，在隔离子进程中运行生成的代码，解析并返回数值结果（β 和 Pf）。
    6.  **Result Interpreter（结果解释器）**：将数值结果转换为工程解释（可靠性等级、与目标值对比、注意事项）。
*   **关键技术细节**：
    *   **结构化数据合约**：智能体间通过 `ProblemSpec`、`AnalysisPlan`、`GeneratedCode`、`ExecutionResult`、`EngineeringInterpretation` 五个预定义数据合约传递信息，保证流程的模块化、可追溯性和鲁棒性。
    *   **人机协同**：用户需在关键决策点（如方法类别确认、具体高级方法选择）进行确认或提供信息，保留非专家用户的控制权。
    *   **后处理方法规划器微调**：算法流程上，先执行所有候选求解器得到参考结果，依据确定性规则（如“当β>3.5时选择Advanced”或“当FORM结果与MCS参考差值<5%时选择FORM-based”）生成标签，然后用此标签微调LLM，使其学会从问题描述“先验”预测合适的类别。
    *   **QLoRA 微调**：针对 Method Planner，使用 QLoRA 策略（4-bit 量化 + 低秩适配器）进行高效微调，仅训练适配器权重。
*   **公式与算法逻辑**：
    *   **目标**：计算失效概率 \(P_f = \int_{g(X) \leq 0} f_X(x) dx\) 和可靠性指标 \(\beta = -\Phi^{-1}(P_f)\)。
    *   **方法选择规则**（用于生成训练标签）：
        *   **优先级1**：若 \(\beta > 3.5\)（稀有失效）→ **Advanced**。
        *   **优先级2**：若 \(\beta \leq 3.5\) 且 FORM 估计的 \(\beta\) 与 MCS 参考 \(\beta\) 之差小于 5% → **FORM-based**。
