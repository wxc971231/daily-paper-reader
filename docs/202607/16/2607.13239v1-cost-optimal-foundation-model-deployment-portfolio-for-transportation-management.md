## 摘要
基础模型（包括大语言模型和视觉语言模型）越来越多地用于交通管理中心任务，如异常检测、事件报告和出行者信息。在TMC各项功能中部署多个此类模型会引发一个组合问题：每个功能应由哪个模型服务、采用何种部署模式、以及在何种共享硬件预算下进行？我们将此问题形式化为基础模型部署组合（FMDP）问题，这是一个混合整数规划，在共享GPU容量下最小化总拥有成本，同时满足每项功能的质量、延迟和安全约束。我们通过从0-1背包问题归约证明该问题是NP难的，并提出一种多项式时间的贪心启发式算法。在一个包含五项TMC功能和19个候选（模型，模式）对的示例案例研究中，FMDP确定了一个混合组合，成本为每月34美元（比最便宜的可行全封闭API基线低97%），方法是将四项功能路由到开源API，而一项没有开源模型满足其质量下限的功能则路由到封闭API。盈亏平衡分析表明，只有当视觉查询速率超过约309次/小时或API价格翻倍时，本地GPU投资才变得合理。

## Abstract
Foundation models, including large language models (LLMs) and vision-language models (VLMs), are increasingly used for transportation management center (TMC) tasks such as anomaly detection, incident reporting, and traveler information. Deploying multiple such models across TMC functions raises a portfolio question: which model should serve each function, in which deployment mode, and under what shared hardware budget? We formulate this as the Foundation Model Deployment Portfolio (FMDP) problem, a mixed-integer program minimizing total cost of ownership (TCO) subject to per-function quality, latency, and safety constraints over shared GPU capacity. We prove the problem NP-hard by reduction from the 0-1 knapsack problem and propose a polynomial-time greedy heuristic. In an illustrative case study with five TMC functions and 19 candidate (model, mode) pairs, FMDP identifies a mixed portfolio costing $34/mo (97% below the cheapest feasible all-closed-API baseline) by routing four functions to open-source APIs and the one function whose quality floor no open-source model meets to a closed API. Break-even analysis shows that on-premise GPU investment becomes reasonable only above approximately 309 vision queries/hour or if API prices double.

- **研究动机**：交通管理中心（TMC）越来越多地使用大语言模型（LLMs）和视觉语言模型（VLMs）执行异常检测、事故报告生成、出行者信息推送等任务。部署多个基础模型面临组合优化问题：每个功能应该选择哪个模型、采用哪种部署模式（闭源API、开源API、云端GPU、本地GPU）、在共享GPU预算下如何分配以实现总拥有成本（TCO）最小化，同时满足质量、延迟和安全约束。
- **背景**：现有工作（如FrugalGPT、RouteLLM）仅针对单任务优化，未考虑多任务共享基础设施的相互影响；而TMC需要同时运行多项功能，不同模型在质量、成本、安全性上差异巨大，且硬件资源有限。论文首次将这一问题形式化为**基础模型部署组合（FMDP）问题**。

## 2. 论文提出的方法论：核心思想、关键技术细节
- **核心思想**：将TMC的功能部署决策建模为混合整数规划（MIP），在共享GPU容量和功能级约束下最小化月度TCO。
- **关键技术细节**：
  - **部署模式**：定义四种模式：闭源API、开源API、云端GPU、本地GPU。每种模式对应不同计费方式（按token、按小时、按设备摊销等）。
  - **成本模型**：总成本 = API token费用 + 固定硬件成本（GPU租赁或摊销） + 可变能源成本（仅本地GPU）。GPU模式下，加载一个模型需占用n=⌈mem/cap⌉设备，且仅当有功能使用时才激活。
  - **约束条件**：
    - 每个功能只能分配一个（模型，模式）对（式5）；
    - 质量得分不低于下限（式6）；
    - 安全指标（如错误率）不超过上限（式7）；
    - 每个功能延迟不超过上限（式8）；
    - 若功能使用GPU模式，则对应模型需加载（式9）；
    - 总GPU使用量不超过各模式可用数量（式10）。
  - **复杂度**：证明FMDP是NP-hard（通过从0-1背包问题归约）。
  - **算法**：提出多项式时间的两阶段贪心启发式算法：
    - **阶段一（最小成本分配）**：按任意顺序遍历功能，为每个功能分配当前最便宜的可行对（考虑已加载模型可共享硬件降低成本）。
    - **阶段二（容量修复）**：若GPU预算超限，对每个GPU分配的功能计算切换到API的**成本增量Δf**和**释放GPU设备数gf**，选择修复比rf=Δf/gf最小的功能进行切换，重复直到满足预算。若所有rf=+∞（无设备释放），则选择Δf最小的功能切换。
  - **复杂度**：O(F·|A|_max + F²)，其中F为功能数，|A|_max为最大候选对数量。
