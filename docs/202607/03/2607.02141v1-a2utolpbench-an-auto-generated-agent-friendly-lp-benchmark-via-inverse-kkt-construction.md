## 摘要
大多数从文本中提取线性规划的基准测试都是手工编写和标注的静态单词问题数据集。一旦这样的数据集发布，其大小固定，难度固定，每个问题都可能泄露到未来大语言模型的训练数据中。我们提出了\textbf{A$^{2}$utoLPBench}，一个用于测试LLM驱动的智能体在纯文本线性规划问题上的基准测试。我们首先选择一个可行点和其对偶，然后写出一个问题，使得该点是最优解且目标值已知。答案通过构造已知，无需求解器调用或人工标注。评估环境包含一个参考求解器-批评者基线和一个Docker镜像，其使用说明是为LLM驱动的智能体读取而编写的。有了这些，任何智能体都可以通过一条命令运行基准测试并获得校准分数。由于该基准测试是一个生成器，而非固定数据集，它具有任何固定数据集无法匹敌的特性：无限的新鲜问题供应、由$(n,m)$设置的难度旋钮、通过构造正确的真实答案、相对于人工编写每个问题的低LLM侧成本、跨独立批次的可重复分数，以及在使用新的事后截止种子范围时对训练数据泄漏的抵抗力。

## Abstract
Most LP-from-text benchmarks are static datasets of word problems written and labeled by hand. Once such a dataset is released, its size is fixed, its difficulty is fixed, and every problem can leak into the training data of future LLMs. We present \textbf{A$^{2}$utoLPBench}, a benchmark for testing LLM-driven agents on linear programming problems written in plain text. We first pick a feasible point and dual, then write down a problem for which that point is optimal and the objective value is known. The answer is known by construction, with no solver call and no human annotator. The evaluation environment bundles a reference solver-critic baseline and a Docker image whose usage instructions are written for an LLM-driven agent to read. With these in place, any agent can run the benchmark and get a calibrated score with one command. Because the benchmark is a generator rather than a fixed dataset, it has properties no fixed dataset can match: an unlimited supply of fresh problems, a difficulty knob set by $(n,m)$, ground-truth answers correct by construction, low LLM-side cost per problem relative to human authoring, repeatable scores across independent batches, and resistance to training-data leakage when fresh post-cutoff seed ranges are used.

- **本文目标**：设计一个**生成器型基准**，可无限产生新鲜、防泄漏、难度可调、且答案数学可证明正确的LP问题，并附带智能体友好的一键运行环境。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想：逆KKT构造
- 不是先随机生成问题再求解得到答案，而是**先选取一个可行解和对偶变量，然后反向推导出问题**，使该点根据KKT条件自动成为最优解，目标值即已知。
- 具体方法：
  1. 随机采样原始可行点 \(x > 0\)（整数）、系数矩阵 \(A\)（确保满秩）、对偶变量 \(\lambda > 0\)（实数）。
  2. 通过公式构造：
     - \(b = Ax\) （使所有原始约束在 \(x\) 处紧）
     - \(c = A^\top \lambda\) （使对偶可行性等式成立）
     - 目标函数最优值 \(\phi = c^\top x\)
  3. 根据KKT定理，这样构造的 \((A,b,c)\) 与对偶变量自动满足所有KKT条件，因此 \(x\) 是最优解，\(\phi\) 是真实最优值，**无需调用任何求解器，也无需人工验证**。

### 生成器组件（Auto部分）
- 暴露参数旋钮：变量数 \(n\)、约束数 \(m\)、系数范围、随机种子。
- 通过单次LLM调用将数学三元组转换成自然语言描述（NL Drafter）。
- 支持任意维度和种子范围，可无限生成新实例。

### 智能体组件（Agent部分）
- **求解器-批评者双智能体基线**：包含三个动作：
  - Propose：求解器根据问题描述生成初始Python代码。
  - Audit：批评者检查代码及其输出，给出“AGREE”或“DISAGREE”及理由。
  - Refine：若批评者不同意且迭代未达上限，求解器根据反馈修正代码。
  - 最多执行 \(K=3\) 轮迭代。
- **Docker镜像**：打包生成器、验证器、双智能体运行时和AGENTS.md说明书，任何支持工具调用的LLM智能体可通过一条 `docker run` 命令接入并获得校准分数。

## 6. 论文的主要结论与发现

1. **生成器型基准的优越性**：A$^{2}$utoLPBench可无限生成新鲜实例，避免数据泄露，且通过\((n,m)\)平滑调节难度，这是静态数据集无法做到的。
2. **难度曲线可测量**：在自建基准上，vanilla求解器成功率从饱和区（100%）骤降至40×40层的8.3%（DeepSeek-V4），展示了清晰的“能力悬崖”。
3. **求解器-批评者循环提升显著**：在MAMO complex上，跨模型配对最多带来+15.1pp（Kimi-K2.5）和+11.3pp（Claude-Sonnet-4.6）的提升，而饱和基准（NL4OPT）上无提升甚至微降（同一分布批评者有时过度质疑）。
4. **批评者配对依赖基准**：当失败类型是公式化错误时，跨模型批评者更有效；当是数值精度问题时，同模型批评者更可靠。
5. **OpenAI-GPT5.4例外**：在禁用推理模式下，GPT5.4的Refine能力弱，导致A$^{2}$LPBench模式反而小幅下降（-1.9pp），表明循环有效性依赖于求解器的纠错能力。

- **难度可编程**：通过\((n,m)\)和系数范围精细控制难度，可追踪不同规模下的能力曲线。
- **智能体友好**：完整Docker环境+AGENTS.md+标准工具接口，降低评估门槛。

## 8. 不足与局限

- **自然语言渲染的多样性有限**：当前拖稿器偏向通用OR场景（资源分配、生产计划等），缺乏领域多样性（供应链、能源等）。
- **拖稿器布局可能引入额外难度**：不同拖稿器模型产生不同的矩阵/右端项排版方式，可能影响求解器解析难易，但未控制。
- **外部基准实验采样局限**：只用了前53个问题，未使用随机子样本，可能无法代表整个基准分布。
- **部分求解器仅单批测试**：Qwen3.5和Kimi-K2.5的校准曲线只有一批，统计可靠性不足。
- **求解器-批评者循环的局限性**：当求解器Refine能力弱时，循环可能无效甚至降级（如GPT5.4）；同一分布批评者可能产生假阳/假阴。
- **缺乏消融研究**：未单独分析拖稿器、批评者提示、LP建模教程等因素的影响。
- **禁用推理模型thinking模式**：可能限制了Kimi-K2.5、GPT5.4等模型在复杂问题上的表现，使对比偏向于“直接生成”模式。
- **数值精度依赖**：虽然KKT保证解析最优值，但自然语言拖稿器要求系数逐字呈现，无精度损失；但LLM生成的代码求解可能因数值问题出现误差（s40x40层失败主要为数值原因）。
